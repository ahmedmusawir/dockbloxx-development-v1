## Lessons Learned

**From Case Study #1 (axios removal):**
- `npm audit` advisories are a usage-audit trigger, not just a patch-in-place trigger. Sometimes the right answer is *removal*, not upgrade.
- A dependency declared in `package.json` but referenced only in docs is a maintenance trap — CVE risk with zero functional benefit.
- The headline vuln count is a lagging indicator: `npm audit fix` had already patched axios, but removing it eliminated 18 CVEs from the dep graph going forward.
- Aspirational doc examples (the `frontend.md` "Alternative HTTP Client" section) are a strong signal of unused deps. **Always grep `*.md`, not just source.**

**From Case Study #2 (swiper@11→12 migration):**
- Documented upstream breaking changes don't always *manifest* as visible changes in any given codebase. v12's SVG-nav-arrow swap was the flagged risk and it rendered cleanly with no restyling needed. The scope of *your* usage determines real impact, not the changelog severity.
- Small usage surfaces flip the cost calculus: migrating a dep used pervasively might justify staying on a patched older major; migrating a dep used in one component is faster than maintaining a CVE-acceptance rationale.
- Bundle-size delta is a useful sanity check on dep upgrades. +0.5 kB on `/shop/[slug]` after swiper@12 is fine; a 50 kB delta would have been a red flag.
- **Tony's pause-at-step-4 modification** for customer-facing UI upgrades is now codified in the playbook as **P7** (Require visual sign-off before finalizing customer-facing upgrades). 2-minute cost on a money-path UI; locking in "no regression" prematurely is the real risk.

**Workflow-level lesson (both case studies):**
- `npm audit fix --force` should never be reflexive. It proposed downgrading **Next 15 → 9.3.3** during this session — a vastly worse outcome than the moderate postcss CVE it was supposedly fixing. Always inspect the proposed change before running `--force`.
