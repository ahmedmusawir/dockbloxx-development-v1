**Lessons Learned**

- Sometimes the "upgrade to fix" path doesn't actually fix anything â€” always verify the bundled transitive version of the would-be-upgrade target *before* committing to the upgrade. (`npm view <pkg>@<version> dependencies.<dep>` is the one-liner.)
- npm `overrides` requires the direct dep range to intersect the override range; if you get `EOVERRIDE`, bump the direct dep to match before retrying. This is a non-obvious gotcha for first-time `overrides` users.
- A "moderate XSS via CSS stringify" advisory in a build-time-only dep is a CVE-hygiene play, not a real-risk fix â€” but doing it cleanly via overrides costs ~10 min and gives a clean audit posture for any future security gate, so it's worth doing.
- The override applied to **every** nested postcss instance in the dep tree, including Next's. npm's override resolution is more powerful than people sometimes assume â€” it's not just for top-level deps.
