## Lessons Learned

- **`acf` key in GET response ≠ write support.** The native WP REST `acf` integration enables READS but does not reliably handle repeater WRITES. On dockbloxx.mystagingwebsite.com specifically, the `POST /wp/v2/pages/{id}` with `{acf:{dealer_data:[...]}}` was silently accepted (200 OK) and silently no-op'd (no rows written, `_acf_changed: false`). For any future script-driven ACF seed against this install, confirm the `/wp-json/acf/v3/...` routes exist first (i.e. ACF to REST API plugin installed) before designing the request shape — or fall back to WP-CLI.
- **Attribute observed data carefully.** When the initial GET showed dealer 1 byte-perfect after the script ran, I leapt to "partial write succeeded" — but Tony had placed that exact row manually as a test before running the script. With only the post-run GET to look at, both hypotheses (partial write vs. pre-existing test row) fit equally; I should have asked Tony to confirm whether the pre-run page state was empty before diagnosing. Lesson: when the failure mode is "silent no-op," establishing the pre-state is required, not optional.
